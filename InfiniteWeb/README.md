InfiniteWeb automatically synthesizes fully functional web environments for GUI agent training. Given a website seed (a website type description and a design image), it generates multi-page websites with realistic data, business logic, user tasks, and dense reward evaluators — all without any manual effort.

## 📢 News

- 🎉 **2026.03** — Code, dataset, and generated websites are now publicly available! Explore our generated websites at [infiniteweb-ai.github.io](https://infiniteweb-ai.github.io/).
- 📄 **2026.01** — Paper released on [arXiv](https://arxiv.org/abs/2601.04126)!

## ✨ Key Features

- 🚀 **End-to-End Automation** — From a simple seed to a fully functional website, no manual coding needed
- 🌐 **Realistic & Interactive** — Generates real data, working APIs, and business logic, not static mockups
- 🎯 **Task & Evaluator Generation** — Automatically creates user tasks with dense reward evaluators for RL training
- 🎨 **Design-Guided Styling** — Matches the visual style of a reference design image across all pages
- 📦 **Scalable Batch Generation** — Generate hundreds of diverse websites in parallel
- 🔗 **OSWorld Compatible** — Fully compatible with [OSWorld](https://github.com/xlang-ai/OSWorld), deploy and evaluate in 4 steps
- 🧪 **TDD Validation & Auto-Fix** — Auto-generated tests with iterative fix loops ensure functional correctness
- 🏆 **Dense Reward Evaluators** — Checkpoint-based 0.0–1.0 scoring with anti-reward-hacking design for effective RL training

## 🚀 Getting Started

### Step 1: Prerequisites

```bash
conda create -n infiniteweb python=3.10 nodejs -y
conda activate infiniteweb
pip install -r requirements.txt
```

### Step 2: Configuration

1. Copy the template config and fill in your credentials:

```bash
cp config/config_template.json config/my_config.json
```

2. Edit `my_config.json` and replace the placeholders:

| Placeholder | Description |
|-------------|-------------|
| `YOUR_AZURE_OPENAI_ENDPOINT` | Your Azure OpenAI endpoint URL (e.g., `https://your-resource.openai.azure.com/`) |
| `YOUR_PEXELS_API_KEY` | *(Optional)* [Pexels API](https://www.pexels.com/api/) key for real image replacement |
| `YOUR_FREESOUND_API_KEY` | *(Optional)* [Freesound API](https://freesound.org/docs/api/) key for audio resources |
| `YOUR_YOUTUBE_API_KEY` | *(Optional)* [YouTube Data API](https://developers.google.com/youtube/v3) key for video resources |
| `YOUR_GOOGLE_API_KEY` | *(Optional)* Google API key for Custom Search |
| `YOUR_GOOGLE_CSE_CX` | *(Optional)* Google Custom Search Engine ID |

### Step 3: Generate Websites

**Single website:**

```bash
# Custom website type with design image
python src/tdd_generator.py --config config/my_config.json \
    --website-type "online_bookstore_website" \
    --design-image "resource/example.jpg"
```

**Batch generation:**

```bash
# Generate from a websites config file
python src/batch_generate.py \
    --config config/my_config.json \
    --websites-config config/website_seeds_template.json \
    --concurrent 3

# Resume from a previous batch run
python src/batch_generate.py \
    --config config/my_config.json \
    --resume results/batch_generated/20251204_120000 \
    --websites-config config/website_seeds_template.json
```

### Step 4: Generate OSWorld-Compatible Task JSONs (Optional)

InfiniteWeb leverages [OSWorld](https://github.com/xlang-ai/OSWorld)'s infrastructure — all generated websites and tasks can be directly deployed in OSWorld for agent testing and training. For detailed integration instructions, see the [OSWorld Integration Guide](osworld_integration/README.md).

```bash
python src/generate_task_jsons.py --batch-dir results/batch_generated/20251204_120000 --template osworld_integration/task_template.json
```

### Step 5: Functional Testing with ReAct Agent (Optional)

After generation, you can optionally run an end-to-end web agent to functionally test each website and filter tasks by quality. The agent operates a browser via `playwright-cli` and attempts to complete each task, providing pass/fail results for task selection.

```bash
# Prerequisites
npm install -g @playwright/cli

# Test a single website
python src/react_test_runner.py \
    --website-dir results/generated \
    --config config/my_config.json

# Test a batch of websites
python src/react_test_runner.py \
    --batch-dir results/batch_generated/20251204_120000 \
    --config config/my_config.json \
    --concurrent 5

# Test a specific task
python src/react_test_runner.py \
    --website-dir results/generated \
    --config config/my_config.json \
    --task-id "task_1"
```

## ⚙️ Configuration Options

Key options in the config file:

| Option | Description |
|--------|-------------|
| `output_dir` | Output directory for generated websites |
| `deployment` | Model deployment name (e.g., `gpt-5`) |
| `endpoints` | Azure OpenAI endpoint URLs |
| `max_concurrent` | Maximum concurrent LLM calls |
| `max_pages` | Maximum number of pages per website |
| `max_data_items` | Maximum data items to generate |
| `task_count_range` | Range of tasks to generate (e.g., `"8-10"`) |
| `image_mode` | `"Real"` for API-sourced images, `"Placeholder"` for placeholders |
| `skip_tdd_validation` | Skip TDD test validation step |
| `stage_configs` | Per-stage model and reasoning effort overrides |

## ❓ FAQ

<details>
<summary>Where is the data stored?</summary>

All website data is stored in the browser's `localStorage`. The data is automatically injected when the user first visits the `index.html` page, so no external database or server is required.
</details>

<details>
<summary>Where does the backend run?</summary>

There is no separate backend server. All backend logic is implemented in `business_logic.js`, which runs entirely in the browser. Each HTML page imports this file and calls its functions directly, making every generated website fully self-contained and static.
</details>

<details>
<summary>Why does the second website still show data from the first one?</summary>

Because `localStorage` is shared across pages served from the same origin, opening a different website without clearing the storage will show stale data. You need to manually call `localStorage.clear()` in the browser console before switching websites. During training in OSWorld, we automatically call the cleanup function before each task.
</details>
